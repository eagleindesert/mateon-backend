## 🆕 v6 → v6-2 변경 요약

이번 개정에서는 **활동(Event) 도메인의 트래픽 과부하 방지 및 조회 성능 최적화(페이지네이션)**와 **`category`·`field` DB NOT NULL 제약 전환 및 카테고리 확충** 변경 사항이 추가되었습니다.

* **🆕 활동 DB 컬럼 `category` 및 `field` NOT NULL 제약 전환 (V21 마이그레이션)**:
  * `events.category` 및 `events.field` 컬럼에 DB 수준의 `SET NOT NULL` 제약을 적용했습니다.
  * `EventRequestDTO.field`에 `@NotNull` 검증 어노테이션이 추가되어, 등록 시 `field` 누락 시 DB 위반(500) 대신 `400 Bad Request` 응답을 반환합니다.
  * `Event.Category` enum에 `ETC`(기타) 항목이 새로 추가되었습니다.
* **🆕 활동 검색 API 오프셋 페이지네이션 도입**: `GET /api/events/search`
  * `page` (기본값 `0`), `size` (기본값 `20`, 최대 `100`) 쿼리 파라미터가 추가되었습니다.
  * DB 수준(`JpaSpecificationExecutor` + `Pageable`)에서 필터링, 정렬(`startDate` 최신순), 페이징을 일괄 처리합니다.
  * 과도한 `size` 요청으로 인한 서버 과부하를 방지하기 위해 `MAX_PAGE_SIZE = 100` 상한 제한 및 값 보정(`clampPage`/`clampSize`)을 적용합니다.
* **🆕 전체 활동 무작위 조회 API 표본 크기 제한**: `GET /api/events`
  * `size` (기본값 `20`, 최대 `100`) 쿼리 파라미터가 추가되었습니다.
  * `ORDER BY RANDOM()`의 특성상 offset 페이징 대신 random 표본 크기 제한(`LIMIT :size`)을 두어 응답 크기만 제한합니다.
* **프론트엔드 호환성 유지**:
  * 기존 클라이언트의 호환을 유지하기 위해 응답 `data`는 순수 배열(`List<EventResponseDTO>`) 구조를 그대로 유지합니다.
  * 다음 페이지 존재 여부는 반환된 데이터 개수가 요청한 `size`와 동일한지 확인하여 추정할 수 있습니다.

## 🆕 v5 → v6 변경 요약

이번 개정은 전부 **활동(Event) 도메인**에 집중되어 있습니다. 팀/알림/채팅/매칭 관련 API는 변경 사항이 없습니다.

* **🆕 활동(공모전 등) 등록 API 추가**: `POST /api/events` — 그동안 활동은 크롤러/수동 SQL로만 들어왔는데, 이제 API로 직접 등록할 수 있습니다.
  * **인증 필수**(비로그인 403 `FORBIDDEN_ACCESS`, SecurityConfig에 `POST /api/events` 전용 매처가 추가됨 — 다른 `/api/events/**` GET 경로는 계속 비로그인 허용).
  * 중복 검사는 하지 않습니다. 같은 공모전을 두 번 등록하는 것을 서버가 막지 않으며, 이는 의도된 동작입니다(아래 `external_id` 제약 변경 참고).
  * 상세 요청/응답 필드는 하단 "4. 일정/활동 (Events)" 섹션 참고.
* **🆕 활동 분야(Field) 분류 추가**: 활동의 '종류'(`category`: `CONTEST`/`EXTERNAL`/`SCHOOL`/`ETC`)와는 별개 축으로, 공모전 사이트의 분야 필터 칩에 대응하는 `field`가 신설되었습니다.
  * `Event.field`, `EventRequestDTO.field`(등록 시 선택), `EventResponseDTO.field`/`fieldLabel`(한글 표기, 서버가 계산해 함께 내려줌).
  * 값은 21개 고정 목록이며 DB `CHECK` 제약으로 오타를 막습니다. 목록은 하단 Enum 정의(`Event.Field`) 참고.
  * 공고 하나에 분야가 여럿이면 분야마다 행을 나눠 등록하고 같은 `externalId`를 공유시키는 방식을 씁니다(컬럼이 값 하나만 담을 수 있어서).
  * 기존 데이터는 분야를 알 수 없으므로 `field`/`fieldLabel` 모두 `null`일 수 있습니다.
* **🆕 활동 검색에 `school`(대학교), `field`(분야) 필터 추가**: `GET /api/events/search`
  * `school` — 대상 대학교 자유 입력 문자열, 부분일치(LIKE). "전체"를 보내면 필터 미적용으로 취급합니다.
  * `field` — `Event.Field` enum 값으로 정확히 필터링합니다.
  * 기존 `college`(단과대) 파라미터는 **deprecated**되었습니다. 대상 범위를 `targetSchool`로 일원화하는 과정이며, 아직 계속 받기는 하지만 신규 클라이언트는 `school`을 쓰세요. 폐기 신호는 별도 응답 헤더 없이 **서버 로그로만** 남습니다(파라미터 하나만 폐기 대상이라 `recommended`처럼 Deprecation 헤더를 붙이지 않았습니다).
* **⚠️ 활동 검색 정렬 방식 변경**: 로그인 여부와 무관하게 이제 **항상 활동 시작일(`startDate`) 최신순**으로 내려갑니다(같으면 id 내림차순, 즉 나중에 등록된 것이 앞).
  * 예전에는 로그인한 사용자에게 관련도 점수(`EventMatchingService`)순으로 정렬해 줬지만, 이 점수는 공고문에 희망직무 문자열이 그대로 들어있어야 붙는 어휘 매칭이라 순위를 신뢰할 수 없었습니다(예: "서버 운영자"는 0점인데 "AI"↔"email" 같은 부분 문자열 오탐은 만점). 신뢰할 수 없는 순서보다 설명 가능한 순서가 낫다고 판단해 제거했습니다.
  * 필터링·정렬이 전부 DB(Specification)로 옮겨가서, 예전처럼 전체 활동을 메모리에 올려 사용자별로 다시 줄 세우는 구조가 아닙니다.
* **⚠️ `GET /api/events/recommended` deprecated 처리**: 위와 같은 이유(관련도 점수를 신뢰할 수 없음)로 이 API 자체가 폐기 예정입니다.
  * 동작은 그대로 유지되지만, 응답에 `Deprecation: true` / `Sunset` 헤더가 추가되고 서버 경고 로그가 남습니다.
  * Sunset 시각: **2026-12-31 23:59:59 UTC** (프론트 전환 일정에 맞춰 조정될 수 있습니다).
  * 대체 방식은 아직 정해지지 않았습니다. 신규 사용은 지양하세요.
* **🆕 활동 등록/검색에 `organizer`(주최/주관), `targetSchool`(대상 대학교) 필드 추가**: 공고에 늘 붙어 나오는 정보인데 담을 컬럼이 없어서, 기존 `campusScope`/`targetColleges`의 의미를 비틀지 않고 새 컬럼을 추가했습니다(전부 additive 마이그레이션).
  * `targetSchool`은 `targetColleges`와 같은 LIKE 부분일치 검색 대상이며, 콤마로 여러 학교를 적을 수 있습니다. 비어 있으면 전국 대상입니다.
  * **`campusScope`/`targetColleges`는 계속 deprecated 상태로 응답에 남아 있습니다** — 이미 읽고 있는 프론트가 있어 응답에서 빼면 기존 화면이 깨지기 때문입니다. 값도 계속 채워집니다.
* **활동 등록 시 `externalId` 제약 완화**: 기존에는 외부 크롤러 식별자용으로 `NOT NULL` + `UNIQUE`였으나, API로 사람이 직접 등록하는 활동에는 대응하는 외부 ID가 없으므로 둘 다 해제했습니다. 손으로 등록할 때는 비워두면 됩니다.
* **🆕 잘못된 enum 값 입력 시 에러 처리 개선**: `category`/`field` 등에 정의되지 않은 값을 보내면 이제 **400** + 허용 값 목록 안내를 받습니다.
  * 이전에는 이런 요청(요청 바디의 enum 파싱 실패, 쿼리 파라미터의 enum 바인딩 실패)이 처리되지 않아 **500 "서버 오류가 발생했습니다"**로 나갔습니다. 클라이언트 입력 오류가 서버 장애처럼 보이는 문제였습니다.
  * 응답 형태는 `@Valid` 실패와 동일한 필드별 메시지 맵입니다(`data.<필드명>`에 `'값' 는 허용되지 않는 값입니다. 가능한 값: ...` 형태로 표기). 상세는 하단 "공통 에러 응답" 섹션 참고.

## 1. 인증 (Auth)

### POST `/api/auth/email/request` — 이메일 인증코드 발송

* **인증 요구사항**: ❌

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|

**Response 200 OK**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`인증코드가 발송되었습니다.`) |

### POST `/api/auth/email/verify` — 이메일 인증코드 검증

* **인증 요구사항**: ❌
* **설명**: 이메일로 발급된 6자리 인증코드를 확인하고, 검증 성공 시 회원가입 API(`/signup`)에 반드시 동반해야 하는 일회용 티켓(`verificationToken`)을 반환합니다.

**Request Body (`EmailVerifyRequest`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| email | String | ✅ | @NotBlank, @Email | 검증할 사용자 이메일 |
| code | String | ✅ | @NotBlank, @Pattern("^\\d{6}$") | 이메일로 받은 6자리 인증코드 |

**Response 200 OK**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`이메일 인증이 완료되었습니다.`) |
| data | Object | 인증 성공 응답 데이터 |
| data.verificationToken | String | 회원가입 시 주체 인증용으로 제출할 일회용 티켓 |

**예외 응답**
* **400 Bad Request**: 인증코드가 일치하지 않거나 만료된 경우 (`INVALID_VERIFICATION_CODE`)

### POST `/api/auth/signup` — 일반 회원가입

* **인증 요구사항**: ❌
* **설명**: 신규 회원 가입을 처리합니다. 회원가입을 완료하기 위해서는 반드시 이메일 인증 완료 후 획득한 일회용 티켓(`verificationToken`)을 포함해야 합니다. 티켓은 발급 후 30분간만 유효하며, 회원가입이 완료되는 시점에 폐기(삭제)됩니다.

**Request Body (`SignupRequest`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| email | String | ✅ | @NotBlank, @Email, .ac.kr 메일 검증 | 가입할 이메일 (교육기관 메일(.ac.kr)만 가능) |
| verificationToken | String | ✅ | @NotBlank | 이메일 인증 완료 후 응답받은 일회용 티켓 |
| password | String | ✅ | @NotBlank, 10자~20자 | 비밀번호 |
| passwordConfirm | String | ✅ | @NotBlank | 비밀번호 확인 |
| name | String | ✅ | @NotBlank, 최대 50자 | 이름 |
| school | String | - | 최대 100자 | 학교명 (예: `단국대학교`) |
| campus | String | - | 최대 50자 | 소속 캠퍼스 (예: `죽전`). 캠퍼스 구분이 없는 학교는 생략 |
| college | String | - | 최대 100자 | 단과대학명 |
| major | String | - | 최대 100자 | 학과명 |
| grade | String | - | 최대 10자 | 학년 (예: `3학년`) |
| interestJobPrimary | String | - | 최대 100자 | 희망 직무 1 순위 |
| interestJobSecondary | String | - | 최대 100자 | 희망 직무 2 순위 |
| interestJobTertiary | String | - | 최대 100자 | 희망 직무 3 순위 |
| tagline | String | - | 최대 200자 | 한 줄 소개 |

**Response 200 OK**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`회원가입이 완료되었습니다.`) |
| data | Object | 토큰 정보 객체 |
| data.accessToken | String | 액세스 토큰 |
| data.refreshToken | String | 리프레시 토큰 |
| data.tokenType | String | 토큰 타입 (`Bearer`) |
| data.expiresIn | Long | 만료 시간 (밀리초) |

**예외 응답**
* **400 Bad Request**:
  - 교육기관 이메일(.ac.kr)이 아닌 경우 (`INVALID_EMAIL_DOMAIN`)
  - 비밀번호 및 비밀번호 확인 불일치 (`PASSWORD_MISMATCH`)
  - 잘못된 티켓을 제출하였거나 30분 유효기간 초과 시 (`INVALID_VERIFICATION_TOKEN`)
  - 이메일 인증이 선행되지 않은 경우 (`EMAIL_NOT_VERIFIED`)

## 4. 일정/활동 (Events)

### 🆕 POST `/api/events` — 활동(공모전 등) 등록

* **인증 요구사항**: ✅ (미인증 시 403 — `SecurityConfig`에서 `POST /api/events`만 별도로 인증을 강제함)
* **설명**: 활동을 API로 직접 등록합니다. 중복 검사는 하지 않습니다 — 같은 공모전을 두 번 등록하는 것을 서버가 막지 않으며, 등록자가 판단할 문제로 둡니다.

**Headers**

| 헤더 | 값 | 필수 |
|---|---|---|
| Authorization | Bearer {accessToken} | ✅ |

**Request Body (`EventRequestDTO`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| category | Category | ✅ | @NotNull | 활동 카테고리 (`CONTEST`, `EXTERNAL`, `SCHOOL`, `ETC`) |
| title | String | ✅ | @NotBlank | 활동 제목 |
| field | Field | ✅ | @NotNull | 🆕 활동 분야. `category`(종류)와는 다른 축 (v6-2부터 필수). 목록은 하단 Enum 정의 참고 |
| description | String | - | - | 활동 상세 본문 |
| imageUrl | String | - | - | 썸네일/포스터 이미지 URL |
| detailUrl | String | - | - | 외부 링크 URL |
| startDate | LocalDate | - | - | 활동 시작일 (YYYY-MM-DD) |

### 🔌 STOMP 웹소켓 연결 및 통신 가이드 (FE 개발자용)

웹소켓을 활용한 실시간 채팅은 STOMP 프로토콜을 사용합니다. 아래 순서대로 연결 및 통신을 진행해주세요.

#### 1단계: 웹소켓 연결 (CONNECT)
* **Endpoint URL**: `ws://{domain}/ws-stomp` (또는 HTTP SockJS 폴백: `http://{domain}/ws-stomp`)
* 연결 시 반드시 JWT 액세스 토큰을 `Authorization` 헤더에 담아 전송해야 합니다. (연결 시점에 서버 인터셉터가 인증을 수행)

**CONNECT 요청 예시 (stompjs 사용)**
```javascript
const client = new StompJs.Client({
  brokerURL: 'ws://localhost:8080/ws-stomp',
  connectHeaders: {
    Authorization: 'Bearer eyJhbGciOiJIUzI...' // 발급받은 액세스 토큰
  },
  debug: function (str) {
    console.log(str);
  },
  onConnect: () => {
    console.log('✅ STOMP 연결 성공!');
    // 연결 성공 후 2단계(구독) 진행
  },
  onStompError: (frame) => {
    console.error('STOMP 에러 발생: ' + frame.headers['message']);
  }
});
client.activate();
```

#### 2단계: 채팅방 구독 (SUBSCRIBE)
* **Destination**: `/topic/room.{roomId}`
* 채팅방에 입장하면 서버로부터 해당 방의 새로운 메시지를 실시간으로 받기 위해 구독(Subscribe)을 설정합니다.

**SUBSCRIBE 요청 및 수신 예시**
```javascript
// roomId가 1인 채팅방 구독
client.subscribe('/topic/room.1', (message) => {
  // 서버로부터 메시지가 브로드캐스트되면 이 콜백이 실행됨
  const receivedData = JSON.parse(message.body);
  console.log('메시지 도착:', receivedData);
});
```

**수신되는 메시지 Payload (`ChatMessageResponse`) 예시**
```json
{
  "messageId": 124,
  "roomId": 1,
  "senderId": 5,
  "senderName": "김단국",
  "content": "안녕하세요! 프로젝트 관련해서 질문 있습니다.",
  "createdAt": "2026-07-14T00:45:12"
}
```

#### 3단계: 메시지 전송 (SEND / PUB)
* **Destination**: `/app/chat.send`
* 구독이 완료된 상태에서 사용자가 메시지를 보낼 때 사용합니다.
* 전송자 정보(senderId)는 서버가 JWT 토큰을 기반으로 자동 식별하므로 클라이언트에서 보낼 필요가 없습니다.

**SEND 요청 예시**
```javascript
const payload = {
  roomId: 1,
  content: "안녕하세요! 프로젝트 관련해서 질문 있습니다."
};

client.publish({
  destination: '/app/chat.send',
  body: JSON.stringify(payload)
});
```

### 🆕 추천 상세 이유 (lazy)

추천 목록의 `label`은 한 줄짜리 짧은 문구입니다. 사용자가 특정 카드를 펼쳐볼 때만 더 긴 AI 설명을 **그 시점에** 받아오도록 분리된 API입니다 (목록의 모든 후보에 대해 미리 생성하면 후보 수만큼 AI 호출이 발생하므로). `GET`이 아니라 `POST`인 이유는 최초 호출 시 AI를 호출해 결과를 캐시에 저장하는 부수효과가 있기 때문입니다.

## 11. 🆕 Matching (제안 문구 초안)

추천받은 팀·유저를 대상으로 **지원/제안 문구 초안을 AI가 미리 써주는** 기능입니다. 흐름상 위치는 다음과 같습니다.

```
GET  /api/matching/recommendations/user-to-team    (추천 목록 조회)
POST /api/matching/recommendations/reason/...       (선택: 상세 이유 보기)
POST /api/matching/proposals/user-to-team            (← 이 섹션: 지원 문구 초안 받기)
     ↓ 사용자가 검토·수정
POST /api/teams/{teamId}/apply                       (실제 지원서 제출 → applicationId 발급)
     (역방향은 POST /api/teams/{teamId}/offers)
```

이 API들은 **아무것도 저장하지 않습니다.** 순수하게 AI가 생성한 텍스트를 미리보기로 돌려줄 뿐이며, 실제로 지원서/제안을 만들려면 반드시 위 기존 API를 별도로 호출해야 합니다. 그래서 응답에는 `id`가 없습니다 — 나중에 `/apply` 또는 `/offers`로 생성되는 `applicationId`/`offerId`가 사실상의 "제안 ID" 역할을 합니다.

### POST /api/matching/proposals/user-to-team — AI 지원 문구 초안 조립 (유저 → 팀)

* **인증 요구사항**: ✅ (로그인 상태)
* **설명**: 내가 추천받은 팀에 지원할 때 쓸 소개(`introduction`)/지원 메시지(`message`) 초안을 AI로 생성합니다.

**Headers**

| 헤더 | 값 | 필수 |
|---|---|---|
| Authorization | Bearer {accessToken} | ✅ |

**Request Body (`ProposalAssemblyRequestDTO`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| teamId | Long | ✅ | @NotNull | 문구 초안을 받을 팀 ID (내 `user-to-team` 추천 결과에 있던 팀이어야 함) |

**Response 200 OK (`ProposalDraftResponseDTO`)**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`성공`) |
| data | Object | 응답 데이터 |
| data.direction | String | `"USER_TO_TEAM"` (호출한 엔드포인트에 따라 서버가 고정값으로 채움) |
| data.teamId | Long | 대상 팀 ID |
| data.userId | Long | 요청한 본인의 유저 ID |
| data.synergyScore | Double | 추천 단계에서 계산된 적합도 점수를 그대로 반환 (여기서 재계산하지 않음) |
| data.summary | String | AI가 작성한 한 줄 요약 초안 — 지원서의 `introduction`에 넣으면 됨 |
| data.message | String | AI가 작성한 본문 초안 — 지원서의 `message`에 넣으면 됨 |

**예외 응답**

  - 팀이 더 이상 존재하지 않는 경우 (`RESOURCE_NOT_FOUND`)
* **502 Bad Gateway**: AI 서버 응답 처리 실패 (`AI_SERVER_ERROR`)
* **503 Service Unavailable**: AI 서버 연결 실패/타임아웃 (`AI_SERVER_UNAVAILABLE`)

> **캐싱 없음**: 위 "추천 상세 이유" API와 반대로, 호출할 때마다 AI가 **매번 새로 문구를 생성**합니다. "이 초안 말고 다른 버전으로 다시 써줘" 하고 같은 요청을 반복하는 것이 정상적인 사용 패턴입니다.

## 공통 에러 응답

`GlobalExceptionHandler`에 의해 핸들링되는 모든 에러는 `ApiResponse.error()` 포맷을 따릅니다.
기존 에러가 400 Bad Request에 편향되어 고정되어 있던 문제점을 수정하여, 예외가 지정하는 HTTP 상태 코드(예: 429 Too Many Requests 등)를 활용하여 정교한 API 제어를 지원합니다.

### 공통 에러 응답 구조

|---|---|---|

### HTTP 상태 코드별 예외 조건

| `400 Bad Request` | `MateonException` | `이미 사용 중인 이메일입니다.`, `교육기관 이메일(.ac.kr)만 사용 가능합니다.`, `인증코드가 올바르지 않거나 만료되었습니다.` 등 |

> 🆕 **v6 변경**: `HttpMessageNotReadableException`/`MethodArgumentTypeMismatchException` 핸들러가 추가되기 전에는 enum 값이 잘못된 요청(예: `category`에 정의되지 않은 값)이 이 두 핸들러를 못 만나 catch-all(`Exception`)로 떨어져 `500 서버 오류가 발생했습니다`가 나갔습니다. 이제 400으로 정상 분류됩니다. 요청 바디/쿼리 파라미터에 enum을 받는 모든 엔드포인트에 공통 적용됩니다(활동 API에 국한되지 않음).

## 학교 / 캠퍼스 (Enum 아님)

전국 대학을 대상으로 하므로 `User.school`, `User.campus`, `Event.campusScope`, `Event.targetSchool`은 **자유 입력 문자열**이다. 값을 열거하지 않으며 서버는 표기를 검증하지 않는다 (길이 제한만 적용).

| 필드 | 예시 | 비고 |
|---|---|---|
| `User.school` | `단국대학교`, `서울대학교` | 학교명 |
| `User.campus` | `죽전`, `천안` | 캠퍼스 구분이 없는 학교는 `null` |
| `Event.campusScope` | `ALL`, `단국대학교`, `죽전` | ⚠️ deprecated. `ALL`이면 학교 구분 없이 전국 대상 |
| `Event.targetSchool` | `단국대학교`, `단국대학교,고려대학교` | 🆕 v6 추가. 비어 있으면 전국 대상. 콤마로 여러 학교 표기 가능(LIKE 부분일치 검색이라 콤마 구분자 자체는 서버가 해석하지 않는다) |

이벤트 추천 점수 계산 시 `campusScope`가 `ALL`이거나 사용자의 `school` 또는 `campus`와 일치하면 5점이 가산된다. 대소문자와 앞뒤 공백은 무시한다.

> **하위호환**: 과거 enum 값이던 `JUKJEON`, `CHEONAN`도 문자열로 그대로 전송·저장할 수 있다. 다만 신규 클라이언트는 `school` + 한글 캠퍼스명 사용을 권장한다. 값 표기가 안정화되면 표준화할 예정이다.

### 🆕 OfferStatus
역제안(팀→유저) 기능에서 사용됩니다. `ApplicationStatus`와 값이 겹쳐 보이지만, **일부러 별도 enum으로 분리**되어 있습니다 — 지원서 취소는 하드 삭제이지만, 제안 취소는 "팀이 이런 제안을 보냈었다"는 기록을 남기기 위해 `CANCELED` 상태로 보존되기 때문입니다.

| 값 | 설명 |
|---|---|
| `PENDING` | 응답 대기 중. 팀장이 취소할 수 있는 유일한 상태 |
| `ACCEPTED` | 유저가 수락 — 즉시 팀원이 됨 |
| `REJECTED` | 유저가 거절 |
| `CANCELED` | 팀장이 응답 전 회수 |
