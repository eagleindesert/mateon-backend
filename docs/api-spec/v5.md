> Base URL: `/`  
> 인증 방식: JWT Bearer Token (`Authorization: Bearer <accessToken>`)  
> Last Updated: 2026-07-21

## 🆕 v4 → v5 변경 요약

* **매칭(Matching) API 신규 추가**: 팀 추천 기능을 위한 API가 새로 생겼습니다.
  * `POST /api/matching/intents/messages`, `GET /api/matching/intents/session`, `POST /api/matching/intents/session/restart` — 대화형으로 사용자의 희망 역할/기술/활동 스타일 등을 추출하는 "매칭 의도 추출" 세션 API.
  * `GET /api/matching/recommendations/user-to-team` — 의도 추출이 완료된 사용자에게 적합도 점수순으로 팀을 추천하는 API. **의도 추출을 먼저 완료해야만 호출 가능**하며(미완료 시 400 `MATCHING_INTENT_REQUIRED`), `limit` 파라미터는 AI 서버가 상위 10건까지만 점수를 매겨 주므로 그보다 크게 요청해도 최대 10건까지만 내려옵니다.
* **팀(Team) API에 `requiredSkills`(요구 기술 스택) 필드 반영**: `POST/PUT /api/teams` 요청 바디와 `GET /api/teams`, `GET /api/teams/{teamId}` 응답에 `requiredSkills` 필드가 있었으나 v4 문서에 누락되어 있었습니다. 이번 개정에서 문서화했습니다 (실제 API 동작 변경은 없음 — 코드 상 이미 존재하던 필드).
* **(내부 구현, API 스펙 영향 없음)** 팀 임베딩 비동기 갱신이 실패할 경우를 대비해 `team_embeddings` 테이블에 `refresh_status`/`last_error` 등 갱신 시도 상태를 기록하도록 개선되었습니다. 클라이언트가 호출하는 요청/응답 스펙에는 변화가 없습니다.
* **역제안(팀→유저) 기능 추가**: 기존 지원 흐름(유저가 지원 → 팀장이 승인)과 **승인 주체가 반대인** 경로가 생겼습니다. 팀장이 AI 추천 유저 목록을 보고 제안을 보내면, 유저가 수락 여부를 결정합니다. **수락하는 즉시 팀원이 되며 팀장의 재승인은 없습니다** (팀장이 이미 고른 사람이므로).
  * `GET /api/matching/recommendations/team-to-user` — 팀을 질의로 삼아 후보 유저를 적합도순으로 추천. **팀장만** 호출 가능(아니면 403). 팀 임베딩이 아직 준비되지 않았다면 400 `TEAM_EMBEDDING_NOT_READY`.
  * `POST /api/teams/{teamId}/offers`, `GET /api/teams/{teamId}/offers`, `DELETE /api/teams/offers/{offerId}` — 팀장이 제안을 보내고, 확인하고, 회수합니다.
  * `GET /api/teams/offers/me`, `PATCH /api/teams/offers/{offerId}` — 유저가 받은 제안을 확인하고 수락/거절합니다.
  * 제안에는 AI 추천 점수·근거(`aiScore`/`aiLabel`)가 **서버에서 스냅샷**되어 함께 저장됩니다 — 프론트가 되보낼 필요도, 되보낸 값이 쓰이지도 않습니다. 추천을 거치지 않고 보낸 제안은 둘 다 `null`입니다 (정상 케이스).
  * **v5 최초 배포 시점에는 이 6건이 총괄표에만 있고 본문 상세 설명이 누락되어 있었습니다. 이번 개정에서 본문을 채워 넣었습니다.**
* **🆕 협업 온도(팀원 상호 평가) 기능 추가 (3건)**: 팀 활동이 끝난 뒤 팀원끼리 서로를 평가하는 기능입니다.
  * `POST /api/teams/{teamId}/complete` — 팀장이 팀 활동 종료를 선언합니다 (연결된 활동의 종료일이 지나면 매일 새벽 3시 자동으로도 종료 처리됩니다).
  * `GET /api/teams/{teamId}/reviews/targets` — 내가 평가해야 할 팀원 목록과 평가 마감 시각(종료 후 14일)을 조회합니다. **누가 누구를 평가했는지는 절대 노출되지 않고**, 오직 "내가 이 사람을 평가했는지"만 내려줍니다.
  * `POST /api/teams/{teamId}/reviews` — 팀원 전원에 대한 평가(1점 이상 5점 이하)를 한 번에 제출합니다. **제출 후 수정·삭제 불가**하며, 평가 마감 시각이 지나면 제출할 수 없습니다.
  * 평가가 누적되면 `collaborationTemperature`(협업 온도)가 계산되어 팀 상세/유저 추천 응답 등에 노출됩니다. **표본이 부족하면 `0`이 아니라 `null`로 내려갑니다** — 이는 결측이 아니라 "아직 평가받은 이력이 적어 비공개" 라는 의도된 설계입니다.
* **🆕 추천 상세 이유(lazy) 기능 추가 (2건)**: 추천 목록의 `label`은 한 줄짜리 짧은 문구인데, 사용자가 특정 카드를 열었을 때만 더 긴 AI 설명 문구를 **그때 가서** 받아오는 API입니다. 목록의 모든 후보에 대해 미리 AI를 호출하면 비용이 크기 때문에 on-demand로 분리했습니다.
  * `POST /api/matching/recommendations/reason/user-to-team` — 유저 시점에서 특정 팀을 추천받은 상세 이유를 조회합니다.
  * `POST /api/matching/recommendations/reason/team-to-user` — 팀장 시점에서 특정 유저를 추천받은 상세 이유를 조회합니다. (팀장 전용, 403)
  * **한 번 생성된 이유는 DB에 캐시되어 재요청 시 같은 문구를 그대로 반환**합니다 (AI 재호출 없음, 문구가 바뀌지 않음). GET이 아니라 POST인 이유는 최초 호출 시 AI를 호출해 캐시에 쓰는 부수효과가 있기 때문입니다.
  * 해당 팀/유저가 **내(또는 우리 팀의) 최근 추천 결과에 없었다면** 404 `RECOMMENDATION_NOT_FOUND`.
* **🆕 최종 제안 조립(AI 지원/제안 문구 초안) 기능 추가 (2건)**: 추천받은 팀·유저를 대상으로 **지원서/제안 문구 초안을 AI가 미리 써주는** 기능입니다. 아무것도 저장하지 않는 순수 미리보기 단계로, 사용자가 초안을 검토·수정한 뒤 실제로는 기존 `POST /api/teams/{teamId}/apply` 또는 `POST /api/teams/{teamId}/offers`로 별도 전송해야 합니다.
  * `POST /api/matching/proposals/user-to-team` — 유저가 추천받은 팀에 보낼 지원 문구 초안을 요청합니다.
  * `POST /api/matching/proposals/team-to-user` — 팀장이 추천받은 유저에게 보낼 제안 문구 초안을 요청합니다. (팀장 전용, 403)
  * 응답에는 추천 단계에서 계산된 적합도 점수(`synergyScore`)를 그대로 실어주며 여기서 재계산하지 않습니다. **호출할 때마다 AI가 새로 문구를 생성**하므로(캐시 없음), "이 초안 말고 다른 버전으로 다시 써줘" 같은 재요청이 정상 사용 패턴입니다. 위 "추천 상세 이유"와 반대되는 캐싱 정책이니 혼동하지 않도록 주의.
  * 이 기능도 해당 팀/유저가 최근 추천 결과에 없었다면 404 `RECOMMENDATION_NOT_FOUND`.
* **학교/캠퍼스를 전국 대상 자유 입력 문자열로 전환**: v4에서 `Campus` enum(`JUKJEON`/`CHEONAN`)으로 고정되어 있던 캠퍼스 구분이, 전국 대학을 지원하기 위해 `User.school`/`User.campus`/`Event.campusScope` 모두 **자유 입력 문자열**로 바뀌었습니다. 하위호환을 위해 기존 enum 값(`JUKJEON`, `CHEONAN`)도 문자열로 그대로 보낼 수 있습니다. 자세한 내용은 하단 "학교 / 캠퍼스 (Enum 아님)" 섹션 참고.
* **문서 버그 수정**: 매칭 의도 세션 상태(`IntentSessionStatus`) 값이 본문에 `PROGRESS, COMPLETED`로 잘못 예시되어 있었습니다. 실제 값은 `IN_PROGRESS`, `COMPLETED`, `ABANDONED`, `EXPIRED` 4가지이며, `PROGRESS`라는 값은 존재하지 않습니다. Enum 정의 섹션에 정식 표를 추가했습니다.
* **문서 구조 정리**: 팀 협업 평가 섹션(6)을 신설하여 팀 매칭(5)과 알림(7) 사이에 배치했고, 이에 따라 이후 섹션 번호가 한 칸씩 밀렸습니다 (알림 7, 채팅 8, 매칭 의도 추출 9, 매칭 추천 10). 매칭 추천(10) 하위에 있던 "제안 문구 초안" 기능은 별도 섹션(11)으로 분리했고, OAuth 디버그는 최종적으로 12번입니다.

## 1. 인증 (Auth)

### POST `/api/auth/email/request` — 이메일 인증코드 발송

* **인증 요구사항**: ❌
* **설명**: 회원가입용 이메일 인증 코드를 전송합니다. 메일 남용 방지를 위한 60초의 재요청 제한(쿨다운)이 적용됩니다.

**Request Body (`EmailRequest`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| email | String | ✅ | @NotBlank, @Email | 인증코드를 받을 이메일 |

**Response 200 OK**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`인증코드가 발송되었습니다.`) |
| data | Object | `null` |

**예외 응답**
* **429 Too Many Requests**: 60초 이내에 동일 이메일로 다시 요청하는 경우 (`EMAIL_REQUEST_TOO_FREQUENT`)

### POST `/api/auth/email/verify` — 이메일 인증코드 검증

* **인증 요구사항**: ❌
* **설명**: 이메일로 발급된 6자리 인증코드를 확인하고, 검증 성공 시 회원가입 API(`/signup`)에 반드시 동반해야 하는 일회용 티켓(`verificationToken`)을 반환합니다.

**Request Body (`EmailVerifyRequest`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| email | String | ✅ | @NotBlank, @Email | 검증할 사용자 이메일 |
| code | String | ✅ | @NotBlank, @Pattern("^\\d{6}$") | 이메일로 받은 6자리 인증코드 |

**Response 200 OK**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`이메일 인증이 완료되었습니다.`) |
| data | Object | 인증 성공 응답 데이터 |
| data.verificationToken | String | 회원가입 시 주체 인증용으로 제출할 일회용 티켓 |

**예외 응답**
* **400 Bad Request**: 인증코드가 일치하지 않거나 만료된 경우 (`INVALID_VERIFICATION_CODE`)

### POST `/api/auth/signup` — 일반 회원가입

* **인증 요구사항**: ❌
* **설명**: 신규 회원 가입을 처리합니다. 회원가입을 완료하기 위해서는 반드시 이메일 인증 완료 후 획득한 일회용 티켓(`verificationToken`)을 포함해야 합니다. 티켓은 발급 후 30분간만 유효하며, 회원가입이 완료되는 시점에 폐기(삭제)됩니다.

**Request Body (`SignupRequest`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| email | String | ✅ | @NotBlank, @Email, .ac.kr 메일 검증 | 가입할 이메일 (교육기관 메일(.ac.kr)만 가능) |
| verificationToken | String | ✅ | @NotBlank | 이메일 인증 완료 후 응답받은 일회용 티켓 |
| password | String | ✅ | @NotBlank, 10자~20자 | 비밀번호 |
| passwordConfirm | String | ✅ | @NotBlank | 비밀번호 확인 |
| name | String | ✅ | @NotBlank, 최대 50자 | 이름 |
| school | String | - | 최대 100자 | 학교명 (예: `단국대학교`) |
| campus | String | - | 최대 50자 | 소속 캠퍼스 (예: `죽전`). 캠퍼스 구분이 없는 학교는 생략 |
| college | String | - | 최대 100자 | 단과대학명 |
| major | String | - | 최대 100자 | 학과명 |
| grade | String | - | 최대 10자 | 학년 (예: `3학년`) |
| interestJobPrimary | String | - | 최대 100자 | 희망 직무 1 순위 |
| interestJobSecondary | String | - | 최대 100자 | 희망 직무 2 순위 |
| interestJobTertiary | String | - | 최대 100자 | 희망 직무 3 순위 |
| tagline | String | - | 최대 200자 | 한 줄 소개 |

**Response 200 OK**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`회원가입이 완료되었습니다.`) |
| data | Object | 토큰 정보 객체 |
| data.accessToken | String | 액세스 토큰 |
| data.refreshToken | String | 리프레시 토큰 |
| data.tokenType | String | 토큰 타입 (`Bearer`) |
| data.expiresIn | Long | 만료 시간 (밀리초) |

**예외 응답**
* **400 Bad Request**:
  - 교육기관 이메일(.ac.kr)이 아닌 경우 (`INVALID_EMAIL_DOMAIN`)
  - 비밀번호 및 비밀번호 확인 불일치 (`PASSWORD_MISMATCH`)
  - 잘못된 티켓을 제출하였거나 30분 유효기간 초과 시 (`INVALID_VERIFICATION_TOKEN`)
  - 이메일 인증이 선행되지 않은 경우 (`EMAIL_NOT_VERIFIED`)

#### 1단계: 웹소켓 연결 (CONNECT)
* **Endpoint URL**: `ws://{domain}/ws-stomp` (또는 HTTP SockJS 폴백: `http://{domain}/ws-stomp`)
* 연결 시 반드시 JWT 액세스 토큰을 `Authorization` 헤더에 담아 전송해야 합니다. (연결 시점에 서버 인터셉터가 인증을 수행)

**CONNECT 요청 예시 (stompjs 사용)**
```javascript
const client = new StompJs.Client({
  brokerURL: 'ws://localhost:8080/ws-stomp',
  connectHeaders: {
    Authorization: 'Bearer eyJhbGciOiJIUzI...' // 발급받은 액세스 토큰
  },
  debug: function (str) {
    console.log(str);
  },
  onConnect: () => {
    console.log('✅ STOMP 연결 성공!');
    // 연결 성공 후 2단계(구독) 진행
  },
  onStompError: (frame) => {
    console.error('STOMP 에러 발생: ' + frame.headers['message']);
  }
});
client.activate();
```

#### 2단계: 채팅방 구독 (SUBSCRIBE)
* **Destination**: `/topic/room.{roomId}`
* 채팅방에 입장하면 서버로부터 해당 방의 새로운 메시지를 실시간으로 받기 위해 구독(Subscribe)을 설정합니다.

**SUBSCRIBE 요청 및 수신 예시**
```javascript
// roomId가 1인 채팅방 구독
client.subscribe('/topic/room.1', (message) => {
  // 서버로부터 메시지가 브로드캐스트되면 이 콜백이 실행됨
  const receivedData = JSON.parse(message.body);
  console.log('메시지 도착:', receivedData);
});
```

**수신되는 메시지 Payload (`ChatMessageResponse`) 예시**
```json
{
  "messageId": 124,
  "roomId": 1,
  "senderId": 5,
  "senderName": "김단국",
  "content": "안녕하세요! 프로젝트 관련해서 질문 있습니다.",
  "createdAt": "2026-07-14T00:45:12"
}
```

#### 3단계: 메시지 전송 (SEND / PUB)
* **Destination**: `/app/chat.send`
* 구독이 완료된 상태에서 사용자가 메시지를 보낼 때 사용합니다.
* 전송자 정보(senderId)는 서버가 JWT 토큰을 기반으로 자동 식별하므로 클라이언트에서 보낼 필요가 없습니다.

**SEND 요청 예시**
```javascript
const payload = {
  roomId: 1,
  content: "안녕하세요! 프로젝트 관련해서 질문 있습니다."
};

client.publish({
  destination: '/app/chat.send',
  body: JSON.stringify(payload)
});
```

## 10. Matching (팀 추천)

### GET /api/matching/recommendations/user-to-team — 유저 맞춤 팀 추천

* **인증 요구사항**: ✅ (로그인 상태)
* **설명**: 로그인 사용자에게 지원할 만한 모집 중인 팀을 적합도 점수 순으로 추천합니다.
  점수와 추천 근거 문구(`label`)는 AI 서버가 계산하며, 백엔드는 후보 팀의 임베딩/메타데이터를
  모아 전달하고 결과를 팀 정보와 합쳐 내려줍니다.
* **선행 조건**: 의도 추출(`POST /api/matching/intents/messages`)이 **완료**되어 있어야 합니다.
  완료 전에는 400 `MATCHING_INTENT_REQUIRED`로 응답합니다.
* **후보 제외 규칙**: 모집 중(`isRecruiting=true`)이고 임베딩이 계산된 팀만 후보이며,
  **내가 팀장인 팀과 이미 지원서를 낸 팀은 제외**됩니다.

**Headers**

| 헤더 | 값 | 필수 |
|---|---|---|
| Authorization | Bearer {accessToken} | ✅ |

**Query Parameters**

| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|---|---|---|---|---|
| eventId | Long | - | - | 특정 활동에 속한 팀만 후보로 제한 (생략 시 전체 모집 중인 팀) |
| limit | Integer | - | 10 | 내려받을 상위 건수 (점수 내림차순). AI 서버가 후보 중 상위 10건까지만 점수를 매겨 돌려주므로, 10보다 크게 요청해도 실제로는 최대 10건만 내려온다 |

**Response 200 OK**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`성공`) |
| data | Array | 추천된 팀 목록 (점수 내림차순, 추천할 팀이 없으면 빈 배열) |
| data[].teamId | Long | 팀 모집글 ID |
| data[].title | String | 모집글 제목 |
| data[].role | Array (String) | 모집하는 역할 리스트 |
| data[].requiredSkills | Array (String) | 요구 기술 스택 (없으면 `null`) |
| data[].promotionText | String | 홍보용 텍스트 |
| data[].characteristic | String | 팀의 작업 성향 및 특징 |
| data[].capacity | Integer | 모집 정원 |
| data[].currentMemberCount | Integer | 확정된 팀원 수 (팀장 포함). `capacity` 도 팀장을 포함한 정원이므로 `currentMemberCount / capacity` 로 그대로 표시하면 된다 |
| data[].eventId | Long | 연결된 활동 ID (자율 프로젝트면 `null`) |
| data[].connectedActivityTitle | String | 연결된 활동 제목 (자율 프로젝트면 `null`) |
| data[].connectedActivitySummary | String | 연결된 활동 AI 한 줄 요약 (자율 프로젝트면 `null`) |
| data[].leaderId | Long | 팀장(리더)의 사용자 ID — `POST /api/chat/rooms/dm` 의 `targetUserId` 로 그대로 사용 가능 |
| data[].recruitmentEndDate | LocalDate | 모집 종료일 (YYYY-MM-DD) |
| data[].score | Double | AI가 매긴 적합도 점수 (정렬 기준) |
| data[].label | String | 추천 근거 문구 (예: `BE 역할을 모집하고 있어요`) |

**응답 예시**

```json
{
  "success": true,
  "message": "성공",
  "data": [
    {
      "teamId": 39,
      "title": "커머스 백엔드 팀",
      "role": ["BE"],
      "requiredSkills": ["Spring Boot", "PostgreSQL"],
      "promotionText": "커머스 서비스를 만드는 팀입니다.",
      "characteristic": "초보 환영",
      "capacity": 4,
      "currentMemberCount": 2,
      "eventId": 1,
      "connectedActivityTitle": "2026 단국대 창업 아이디어 공모전",
      "connectedActivitySummary": "AI로 캠퍼스 문제를 푸는 팀을 찾습니다",
      "leaderId": 6,
      "recruitmentEndDate": "2026-08-17",

      "label": "BE 역할을 모집하고 있어요"
    }
```

**예외 응답**
* **400 Bad Request**: 의도 추출이 완료되지 않은 경우 (`MATCHING_INTENT_REQUIRED` — `먼저 매칭 의도 추출을 완료해주세요.`)
* **502 Bad Gateway**: AI 서버 응답 처리 실패 (`AI_SERVER_ERROR`)
* **503 Service Unavailable**: AI 서버 연결 실패/타임아웃 (`AI_SERVER_UNAVAILABLE`)

> 참고: 추천할 팀이 없는 경우(후보 0건)는 오류가 아니라 **200 + 빈 배열**입니다.
> 이때는 AI 서버를 호출하지 않습니다.

### 🆕 GET /api/matching/recommendations/team-to-user — **[역제안]** 팀 맞춤 유저 추천 (팀장 전용)

* **인증 요구사항**: ✅ (팀장만 가능, 아니면 403 `FORBIDDEN_ACCESS`)

* **후보 제외 규칙**: 매칭 의도 추출을 완료한 유저 중, **현재 팀원(팀장 포함)**, **이미 이 팀에 지원한 유저**, **이미 이 팀의 제안을 받은 유저**는 제외됩니다.

| 헤더 | 값 | 필수 |

| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|---|---|---|---|---|
| teamId | Long | ✅ | - | 추천의 기준이 될 내 팀 ID |
| limit | Integer | - | 10 | 내려받을 상위 건수. `user-to-team`과 동일하게 AI 서버가 최대 10건까지만 점수를 매긴다 |

| 필드 | 타입 | 설명 |

| data[].tagline | String | 한 줄 소개 |
| data[].desiredRoles | Array (String) | 희망 역할 (매칭 의도에서 추출) |

| data[].collaborationTemperature | Number (BigDecimal) | 협업 온도. **평가 표본이 부족하면 `0`이 아니라 `null`** (비공개, 결측 아님) |

### 🆕 추천 상세 이유 (lazy)

추천 목록의 `label`은 한 줄짜리 짧은 문구입니다. 사용자가 특정 카드를 펼쳐볼 때만 더 긴 AI 설명을 **그 시점에** 받아오도록 분리된 API입니다 (목록의 모든 후보에 대해 미리 생성하면 후보 수만큼 AI 호출이 발생하므로). `GET`이 아니라 `POST`인 이유는 최초 호출 시 AI를 호출해 결과를 캐시에 저장하는 부수효과가 있기 때문입니다.

#### POST /api/matching/recommendations/reason/user-to-team — 추천 상세 이유 조회 (유저 → 팀)

* **인증 요구사항**: ✅ (로그인 상태)
* **설명**: 내가 추천받은 특정 팀에 대한 상세 추천 이유를 조회합니다.

**Headers**

| 헤더 | 값 | 필수 |
|---|---|---|
| Authorization | Bearer {accessToken} | ✅ |

**Request Body (`RecommendationReasonRequestDTO`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| teamId | Long | ✅ | @NotNull | 이유를 조회할 팀 ID (내 `user-to-team` 추천 결과에 있던 팀이어야 함) |

**Response 200 OK (`RecommendationReasonResponseDTO`)**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`성공`) |
| data | Object | 응답 데이터 |
| data.reason | String | AI가 생성한 상세 추천 이유 (긴 설명 문구) |

**예외 응답**
* **400 Bad Request**: 매칭 의도 추출이 완료되지 않은 경우 (`MATCHING_INTENT_REQUIRED`)
* **404 Not Found**:
  - 해당 팀이 내 최근 `user-to-team` 추천 결과에 없었던 경우 (`RECOMMENDATION_NOT_FOUND`)
  - 팀이 삭제되는 등 더 이상 존재하지 않는 경우 (`RESOURCE_NOT_FOUND`)
* **502 Bad Gateway**: AI 서버 응답 처리 실패 (`AI_SERVER_ERROR`)
* **503 Service Unavailable**: AI 서버 연결 실패/타임아웃 (`AI_SERVER_UNAVAILABLE`)

> **캐싱**: 한 번 생성된 이유는 저장되어, 같은 팀을 다시 요청하면 **AI를 재호출하지 않고 저장된 문구를 그대로 반환**합니다. 즉 한번 생성된 문구는 이후 바뀌지 않습니다.

## 11. 🆕 Matching (제안 문구 초안)

추천받은 팀·유저를 대상으로 **지원/제안 문구 초안을 AI가 미리 써주는** 기능입니다. 흐름상 위치는 다음과 같습니다.

```
GET  /api/matching/recommendations/user-to-team    (추천 목록 조회)
POST /api/matching/recommendations/reason/...       (선택: 상세 이유 보기)
POST /api/matching/proposals/user-to-team            (← 이 섹션: 지원 문구 초안 받기)
     ↓ 사용자가 검토·수정
POST /api/teams/{teamId}/apply                       (실제 지원서 제출 → applicationId 발급)
     (역방향은 POST /api/teams/{teamId}/offers)
```

이 API들은 **아무것도 저장하지 않습니다.** 순수하게 AI가 생성한 텍스트를 미리보기로 돌려줄 뿐이며, 실제로 지원서/제안을 만들려면 반드시 위 기존 API를 별도로 호출해야 합니다. 그래서 응답에는 `id`가 없습니다 — 나중에 `/apply` 또는 `/offers`로 생성되는 `applicationId`/`offerId`가 사실상의 "제안 ID" 역할을 합니다.

### POST /api/matching/proposals/user-to-team — AI 지원 문구 초안 조립 (유저 → 팀)

* **인증 요구사항**: ✅ (로그인 상태)
* **설명**: 내가 추천받은 팀에 지원할 때 쓸 소개(`introduction`)/지원 메시지(`message`) 초안을 AI로 생성합니다.

**Headers**

| 헤더 | 값 | 필수 |
|---|---|---|
| Authorization | Bearer {accessToken} | ✅ |

**Request Body (`ProposalAssemblyRequestDTO`)**

| 필드 | 타입 | 필수 | Validation | 설명 |
|---|---|---|---|---|
| teamId | Long | ✅ | @NotNull | 문구 초안을 받을 팀 ID (내 `user-to-team` 추천 결과에 있던 팀이어야 함) |

**Response 200 OK (`ProposalDraftResponseDTO`)**

| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 성공 여부 (`true`) |
| message | String | 결과 메시지 (`성공`) |
| data | Object | 응답 데이터 |
| data.direction | String | `"USER_TO_TEAM"` (호출한 엔드포인트에 따라 서버가 고정값으로 채움) |
| data.teamId | Long | 대상 팀 ID |
| data.userId | Long | 요청한 본인의 유저 ID |
| data.synergyScore | Double | 추천 단계에서 계산된 적합도 점수를 그대로 반환 (여기서 재계산하지 않음) |
| data.summary | String | AI가 작성한 한 줄 요약 초안 — 지원서의 `introduction`에 넣으면 됨 |
| data.message | String | AI가 작성한 본문 초안 — 지원서의 `message`에 넣으면 됨 |

**예외 응답**
* **400 Bad Request**: 매칭 의도 추출이 완료되지 않은 경우 (`MATCHING_INTENT_REQUIRED`)
* **404 Not Found**:
  - 해당 팀이 내 최근 `user-to-team` 추천 결과에 없었던 경우 (`RECOMMENDATION_NOT_FOUND`)
  - 팀이 더 이상 존재하지 않는 경우 (`RESOURCE_NOT_FOUND`)
* **502 Bad Gateway**: AI 서버 응답 처리 실패 (`AI_SERVER_ERROR`)
* **503 Service Unavailable**: AI 서버 연결 실패/타임아웃 (`AI_SERVER_UNAVAILABLE`)

> **캐싱 없음**: 위 "추천 상세 이유" API와 반대로, 호출할 때마다 AI가 **매번 새로 문구를 생성**합니다. "이 초안 말고 다른 버전으로 다시 써줘" 하고 같은 요청을 반복하는 것이 정상적인 사용 패턴입니다.

## 공통 에러 응답

`GlobalExceptionHandler`에 의해 핸들링되는 모든 에러는 `ApiResponse.error()` 포맷을 따릅니다.
기존 에러가 400 Bad Request에 편향되어 고정되어 있던 문제점을 수정하여, 예외가 지정하는 HTTP 상태 코드(예: 429 Too Many Requests 등)를 활용하여 정교한 API 제어를 지원합니다.

### 공통 에러 응답 구조
| 필드 | 타입 | 설명 |
|---|---|---|
| success | Boolean | 에러 상태 이므로 항상 `false` |
| message | String | 상세 에러 사유 (클라이언트 노출용 메시지) |
| data | Object | 항상 `null` |

### HTTP 상태 코드별 예외 조건
| HTTP Status | Exception Class | 예시 메시지 (`message` 필드값) |
|---|---|---|
| `400 Bad Request` | `MateonException` | `이미 사용 중인 이메일입니다.`, `교육기관 이메일(.ac.kr)만 사용 가능합니다.`, `인증코드가 올바르지 않거나 만료되었습니다.` 등 |
| `400 Bad Request` | `IllegalArgumentException` | 입력 매개변수 유효성 위반 에러 구체 내용 |
| `400 Bad Request` | `MethodArgumentNotValidException` | `입력값 검증에 실패했습니다.` |
| `400 Bad Request` | `MateonException` | `이메일 인증 정보가 유효하지 않습니다. 인증을 다시 진행해주세요.` (`INVALID_VERIFICATION_TOKEN`) |
| `401 Unauthorized` | `BadCredentialsException` | `이메일 또는 비밀번호가 올바르지 않습니다.` |
| `403 Forbidden` | `MateonException` | `권한이 없습니다.` (`FORBIDDEN_ACCESS`) — 팀장 전용 API를 팀원/제3자가 호출하는 경우 등 |
| `404 Not Found` | `MateonException` | `요청한 리소스를 찾을 수 없습니다.` (`RESOURCE_NOT_FOUND`) 등 |
| `429 Too Many Requests`| `MateonException` | `인증코드 요청은 잠시 후 다시 시도해주세요.` (`EMAIL_REQUEST_TOO_FREQUENT`) |
| `500 Internal Server Error` | `Exception` | `서버 오류가 발생했습니다.` |
| `502 Bad Gateway` | `MateonException` | AI 서버가 비정상 응답을 반환한 경우 (`AI_SERVER_ERROR`) |
| `503 Service Unavailable` | `MateonException` | AI 서버 연결 실패/타임아웃 (`AI_SERVER_UNAVAILABLE`) |

> 위 표는 대표적인 상태코드-예외클래스 매핑이며, 각 엔드포인트별 세부 에러 코드(`DUPLICATE_RESOURCE`, `OFFER_ALREADY_RESPONDED`, `TEAM_NOT_ENDED` 등)는 해당 API 섹션의 "예외 응답"에 개별적으로 기재되어 있습니다.

## 학교 / 캠퍼스 (Enum 아님)

전국 대학을 대상으로 하므로 `User.school`, `User.campus`, `Event.campusScope`는 **자유 입력 문자열**이다. 값을 열거하지 않으며 서버는 표기를 검증하지 않는다 (길이 제한만 적용).

| 필드 | 예시 | 비고 |
|---|---|---|
| `User.school` | `단국대학교`, `서울대학교` | 학교명 |
| `User.campus` | `죽전`, `천안` | 캠퍼스 구분이 없는 학교는 `null` |
| `Event.campusScope` | `ALL`, `단국대학교`, `죽전` | `ALL`이면 학교 구분 없이 전국 대상 |

이벤트 추천 점수 계산 시 `campusScope`가 `ALL`이거나 사용자의 `school` 또는 `campus`와 일치하면 5점이 가산된다. 대소문자와 앞뒤 공백은 무시한다.

> **하위호환**: 과거 enum 값이던 `JUKJEON`, `CHEONAN`도 문자열로 그대로 전송·저장할 수 있다. 다만 신규 클라이언트는 `school` + 한글 캠퍼스명 사용을 권장한다. 값 표기가 안정화되면 표준화할 예정이다.
